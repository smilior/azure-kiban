01 要件定義 · 01/03

Azureアプリ公開基盤の構築

社内研究テーマ「アイデア探求ワーク」の一環として、AI駆動開発で構築したアプリをAzure上で短時間で公開できる基盤(静的・動的の2種)を構築する。

目的

社内研究テーマ「アイデア探求ワーク」の一環として、AI駆動開発で構築したアプリをAzure上で短時間で公開できる基盤を構築する。基盤は次の2つを作り、今後のアプリ公開のテンプレートとして提示できる状態にする。

  • 静的コンテンツ公開用の基盤
  • 動的コンテンツ公開用の基盤(Python・Next.jsのアプリをデプロイする)

ゴール(検証の完了条件)

  • 静的基盤: テンプレートから静的サイトを1本公開できる
  • 動的基盤: PythonとNext.jsのアプリを各1本公開できる
  • 2基盤それぞれで、新規公開1件の所要時間と月額費用を実測し、現行VM運用と比較した結果を提示できる
  • 検証結果をもとに、本番展開の可否を判断できる材料が揃っている

対象業務と利用部門

チーム内のアプリ公開・運用業務。検証フェーズのため、利用は起案者のチームに限定する。

現状の課題

Azure VM(Ubuntu)にDockerを載せ、3種のコンポーネントでアプリを公開している。

  • infra-compose: nginx・PostgreSQL・Redisをまとめた共通基盤
  • ap-auth: M365 Graph APIを使う認証共通基盤。SSOとOAuthのメール送信を担う
  • ap-xxxx: 個別アプリ(複数を管理)。infra-composeとap-authを共通利用し、DBは1インスタンス内でデータベースを分割して使う

Azure VMでの運用は、構成を自由に組める柔軟性と月額費用の低さで優れている。一方でIaaS運用のため次の2点が負担であり、この負担は基盤の数に比例して増える。

  • 新規公開のたびにVM上での初期構築作業が発生し、公開までのリードタイムが伸びる
  • OS・ミドルウェアの脆弱性対応を自チームで担っており、対応が遅れた期間はセキュリティリスクを抱える

効果見込み

現状の実測値が未取得のため仮置きで算出する。仮の数字は検証フェーズで実測し確定する。

  • 新規アプリ1件の公開環境準備: 現状8時間(仮)→ テンプレート適用で1時間(仮)。年6件(仮)の公開で年42時間を削減する
  • インフラ保守(パッチ適用・脆弱性対応): 現状月4時間(仮)→ マネージドサービス移行で月0.5時間(仮)。年42時間を削減する
  • OS・ミドルウェア層の脆弱性対応をAzureのマネージドサービスへ委譲し、対応漏れによるリスク期間をなくす

スコープ外

  • 本番利用と他部門への展開(検証結果をもとに別途判断する)
  • 既存アプリ(ap-xxxx)と既存基盤(infra-compose・ap-auth)の移行作業

概算費用

検証フェーズの費用は確保済み予算20万円の枠内とする。

  • Azure利用料: 月2万円(仮)× 6か月= 12万円
  • 予備(検証中の構成変更・負荷確認分): 8万円

内訳は仮置きであり、使途の細目は要確認とする。
