02 ADR(設計判断) · 03/03

テンプレートの記述形式(Bicep採用)

環境の記述形式がテンプレートの再現性と保守性を決めるため、Bicep・Terraform・手順書方式を比較しBicepを採用する。

状態: 決定済み
日付: 2026-07-05
関連: 0001-select-hosting-pattern.md(P1)、02_requirements.md(前提P-6・合格ライン)

背景

検証の成果物は「テンプレートから環境一式を短時間で用意できる」こと(前提P-6、合格ライン1件1時間以内(仮))。環境を何で記述するかがテンプレートの再現性と保守性を決める。形式の指定はなく、起案者から選定を一任された。

選択肢

  • 案1: Bicep — Azure純正のIaC。ARMテンプレートの簡潔な記法で、状態ファイルの管理が不要
  • 案2: Terraform — 汎用IaC。マルチクラウド対応だが、状態(state)の置き場所・ロック・保護という運用が別途必要
  • 案3: azコマンドの手順書 — 逐次実行の手順書。学習は最小だが、再現性が実行順と手作業に依存する

決定

案1(Bicep)を採用する。テンプレート=Bicepファイル一式+パラメータ+手順書(README)とし、ランナー(notes/cicd.md)から az deployment で実行できる形にする。

理由

  • Azure専用の本件でマルチクラウド対応の価値がなく、Terraformのstate管理という新しい運用を持ち込まずに済む
  • 宣言的で再現性がある。手順書方式の「実行順や環境差で結果が変わる」問題を避けられる
  • az CLIとVS Code拡張に組み込み済みで、1人運用の学習コストが最小
  • App Service・Private Endpoint・Private DNSといった構成要素への追従が純正ゆえに速い
  • 撤去(US-06-S3)もリソースグループ削除+再デプロイで担保でき、検証の作り直しが安全にできる

捨てた案とその理由

  • 案2(Terraform): 技術としては同等以上だが、state管理の運用が増える。会社標準がTerraformに定まった場合に再評価する
  • 案3(手順書方式): 検証初日の学習用途には使えるが、成果物(P-6)に求める再現性を満たさない
