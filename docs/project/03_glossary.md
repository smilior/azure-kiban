# 用語集: Azureアプリ公開基盤(検証フェーズ)

docs/project/02_requirements.md をはじめ他の文書と用語を一致させる。新しい用語が出たら随時この表に追記する。

## 用語一覧

| 用語 | 定義 | コード上の名前 |
| --- | --- | --- |
| アイデア探求ワーク | 社内の研究テーマ制度。本件はその一環の検証プロジェクト。 | - |
| 静的基盤 | 静的コンテンツを公開するAzure上の基盤。本件で構築する2基盤の1つ。 | 未定 |
| 動的基盤 | PythonとNext.jsのアプリを公開するAzure上の基盤。本件で構築する2基盤の1つ。 | 未定 |
| 基盤テンプレート | 基盤を複製して新規アプリを公開するためのひな形一式(構成定義と手順)。 | 未定 |
| 検証アプリ | 基盤の検証のために公開するサンプルアプリ。静的サイト・Python・Next.jsの3本。 | 未定 |
| 現行VM運用 | Azure VM(Ubuntu)+Dockerによる現在のアプリ公開方式。比較対象。 | - |
| infra-compose | 現行VM運用の共通基盤コンポーネント。nginx・PostgreSQL・Redisをまとめて動かす。 | infra-compose |
| ap-auth | 現行VM運用の認証コンポーネント。M365 Graph APIでSSOとOAuthのメール送信を担う。 | ap-auth |
| ap-xxxx | 現行VM運用の個別アプリケーションコンポーネントの総称。 | ap-xxxx |
| 公開所要時間 | 新規アプリ1件を公開できる状態にするまでの作業時間。US-06で実測する。 | - |
| 共有基盤 | 動的基盤のうち複数アプリが共同利用する部分(PostgreSQL・認証)。アプリ追加時に作り直さない。 | 未定 |
| 社内閉域網 | 社内のプライベートネットワーク。基盤と公開URLはこの中に閉じ、パブリック公開しない。 | - |
| SSO | M365アカウントによるシングルサインオン。動的基盤の認証のMVP。 | - |
| セッション | ログイン後の利用者の状態。現行はRedisで管理。新基盤では基盤の認証機能(Easy Auth)が保持する。 | - |
| トークン | SSO認証後に発行されるアクセス資格情報。現行はRedisに格納。新基盤では基盤のトークンストアまたはManaged Identityで扱う。 | - |
| Easy Auth | App Serviceの組み込み認証。アプリの手前でSSO誘導・トークン検証・セッション発行・トークン保管を代行する。 | - |
| Managed Identity | Azureがアプリ自身に発行するパスワード不要の身分証。シークレットを保存せずGraph API等のトークンをその場で取得する。 | - |
| カスタムドメイン | 基盤発行のURLに代えて割り当てる社内のドメイン名。 | - |
| App Service証明書 | Azureが発行・更新を管理するSSL証明書サービス。カスタムドメインのHTTPS化に使う。 | - |
| 社内VNet | Azure上の社内既存の仮想ネットワーク。基盤の全リソースをここに接続する。 | - |
| Private Endpoint(PE) | AzureのPaaSサービスに社内VNet内の私設IPで届く専用の入口。公開側の入口を塞ぎ、閉域前提(P-1)を実現する。 | - |
| Private DNSゾーン | PEの私設IPを名前解決するAzure側のDNS。VNet内では自動で効くが、社内DNSからは別途引き込みが要る。 | - |
| 自動デプロイ | Git pushを起点にビルドから公開までを自動で行う仕組み。手動デプロイは行わない。 | - |
| PostgreSQL | 動的基盤のアプリが使うリレーショナルデータベース。現行構成と同一製品。 | - |
| Redis | 現行構成でセッションとトークンを格納するインメモリストア。新基盤では基盤の認証機能が代行するため原則設けない(必要時のみ追加)。 | - |
| チーム開発者 | 検証アプリを開発しデプロイする起案チームのメンバー。検証中は起案者1名。 | - |
| 基盤管理者 | 基盤テンプレートを整備し費用と稼働を記録する担当。起案者を想定。 | - |
| アプリ利用者 | 公開された検証アプリをブラウザで使うチーム内の利用者。 | - |
