# 設計書雛形(docs-template)

ap-template-v4 を複製した個別アプリ用の設計書一式。`{{プレースホルダ}}` を埋めると各フェーズの設計書が完成する。

## 使い方

1. 本ディレクトリを複製アプリの `docs/` にコピーする
2. フェーズ順(01 → 06)に各ファイルを埋める
3. `{{...}}` を実値に置換する。`(例)` 行は自アプリの内容に置き換えるか削除する
4. 設計書はコードと同一 PR で更新する

技術スタック・CI/CD・命名規約などテンプレート由来の既定事項は記入済み。変更する場合は `02_基本設計/adr/` に新しい ADR を追加する(既存 ADR は編集しない)。

## ファイル一覧

| フェーズ | ファイル | 内容 |
|---|---|---|
| 01 要件定義 | `01_プロジェクト概要.md` | 目的・KPI・ステークホルダー・スコープ外・マイルストーン |
| | `02_ペルソナ・ユーザーストーリー.md` | ペルソナ、ユーザーストーリー、MVP 対象の線引き |
| | `03_機能要件.md` | 機能一覧、受け入れ基準(Given-When-Then)、画面一覧と遷移図 |
| | `04_非機能要件.md` | 性能・可用性の目標値、認証認可の水準、法務の要否 |
| 02 基本設計 | `01_システム構成.md` | 全体構成図・環境・CI/CD(既定記入済み) |
| | `02_用語集.md` | ドメイン用語とコード命名の対応。実装前に必ず埋める |
| | `03_機能設計.md` | 機能ごとの外部仕様(入出力・受け入れ基準) |
| | `04_画面設計.md` | 画面遷移図、共通レイアウト、画面詳細 |
| | `05_DB設計.md` | ER 図と命名規約(論理設計) |
| | `06_API設計.md` | 共通規約と API 一覧 |
| | `07_認可設計.md` | ロール一覧と権限マトリクス |
| | `adr/` | 技術的決定の記録。`0001` は既定スタックの決定済み ADR |
| 03 詳細設計 | `01_処理設計.md` | 複雑なロジックのみ事前設計。それ以外はコードとテストを正とする |
| | `02_テーブル定義.md` | 物理設計。正は Alembic マイグレーション |
| | `03_単体テスト観点.md` | 正常系・異常系・境界値の観点表 |
| 04 テスト | `01_テスト計画.md` | 範囲・環境・合格基準・テストデータ方針 |
| | `02_結合テストケース.md` | インターフェース単位のテストケース |
| | `03_総合テストケース.md` | シナリオ・性能テスト・脆弱性診断 |
| | `04_受け入れテストシナリオ.md` | ユーザーシナリオと実施記録 |
| 05 リリース | `01_リリース計画.md` | デプロイ手順(既定 CI/CD)とロールバック |
| | `02_リリースノート.md` | 新機能・修正・既知の問題 |
| 06 運用 | `01_監視・障害対応.md` | 監視項目・障害レベル・ポストモーテム |
| | `02_バックアップ・メンテナンス.md` | バックアップ・ログ運用・依存更新 |

実装フェーズの設計書はない。実装規約は `.github/instructions/`、完成条件は 03_詳細設計と CI に従う。

## V字モデル対応表

上流工程の成果物が、対応する検証工程の合格基準になる。

| 上流(作る) | 検証(確かめる) |
|---|---|
| 要件定義(機能要件・非機能要件) | 受け入れテスト・総合テスト |
| 基本設計(機能・画面・DB・API・認可) | 結合テスト |
| 詳細設計(処理・テーブル定義) | 単体テスト |

## コードが正となるもの

設計書と実装が矛盾した場合、下表の実装を正とし、設計書を追従させる。

| 対象 | 正(Single Source of Truth) | 確認方法 |
|---|---|---|
| API 仕様 | FastAPI の自動生成 API ドキュメント | `{{APP_ROOT_PATH}}/docs`(Swagger UI) |
| テーブル定義 | Alembic マイグレーション(`alembic/versions/`) | マイグレーションファイルとモデル定義 |
| 画面 | テンプレート実装(`app/templates/`) | 各画面の HTML / コンポーネント |
