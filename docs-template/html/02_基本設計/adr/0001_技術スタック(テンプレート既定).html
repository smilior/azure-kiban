02 基本設計 · ADR-0001

ADR-0001 技術スタック(テンプレート既定)

記入ガイド

本ファイルは編集しない。覆す場合は新しいADRを起こし、本ADRのステータスを Superseded にする側で参照する。

ステータス

Accepted

日付

テンプレート既定(ap-template-v4 由来)

背景

複製アプリごとに基盤技術を再選定するとAIの命名・実装の揺れと重複判断が生じる。テンプレート ap-template-v4 が採用済みの構成を既定として固定し、個別アプリはドメイン実装に集中する。

決定

以下をテンプレート標準構成として採用する。

分類採用技術
WebフレームワークFastAPI
画面構成Jinja2 SSR + Alpine.js + Tailwind CSS
ORM / マイグレーションSQLAlchemy + Alembic
データベースPostgreSQL デュアルDB(App DB + 外部認証DB ap-safedx 参照専用)
セッション管理Redis
認証共通認証基盤 ap-safedx 連携(任意)。MSystemUser は外部DB参照
実行基盤Docker Compose(共有ネットワーク shared_backend)
CI/CDGitHub Actions + Self-Hosted Runner(develop→STG / main→PRD、push トリガーで自動CD)

検討した代替案と却下理由

代替案却下理由
SPA(React/Vue)+ REST API 分離SSR + Alpine.js より構成が重く、テンプレート用途では過剰
単一DB構成認証情報を各アプリで複製することになり、共通認証基盤 ap-safedx の一元管理を損なう
セッションをDB/JWTで保持失効制御と水平展開で Redis が優位
マネージドCI(GitHub-hosted runner)社内ネットワーク・秘密情報管理の都合で self-hosted を採用

影響・制約

  • 上記構成を前提に他の設計書・スキルが記述される。個別に覆すと整合が崩れる。
  • 覆す判断は新規ADR(連番_タイトル.md)で記録し、本ファイルは変更しない。
  • 外部認証DB ap-safedx はスキーマ変更しない。Alembic 対象は App DB のみ。
