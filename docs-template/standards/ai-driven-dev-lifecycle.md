# AI駆動開発のライフサイクルとドキュメント戦略

AI駆動開発における各フェーズの成果物、ISO対応、TDDの進め方、テストレベル(単体・結合・総合)の位置づけ、テスト用DBの方針をまとめたガイド。

---

## 1. 基本原則

AIがコードを書く分、人間が残すべきは **「意図」と「判断根拠」**。

1. **ドキュメントはリポジトリ内の Markdown に置く**
   `docs/` 配下でコードと一緒にバージョン管理する。AIエージェントがそのまま読めるため、仕様書が毎回のプロンプトの代わりになる。
2. **「What」よりも「Why」を残す**
   コードが何をしているか(What)はAIがコードから読み取れる。復元できないのは「なぜこの技術を選んだか」「なぜこの案を捨てたか」。ADR(Architecture Decision Record)として蓄積する。
3. **二重管理をやめ、生成できるものは生成する**
   API仕様は OpenAPI 定義を正とする。詳細設計書の大部分はテストコードと型定義に語らせる。陳腐化したドキュメントはAIのコンテキストを汚染するため、更新しないと決めたものは削除する。

## 2. フェーズ別に残すべき資料

### 2.1 企画
- **企画書(PRD)**: 背景・課題、ターゲットユーザー、提供価値、成功指標(KPI)、スコープ外の明記。全フェーズでAIに渡す「北極星」。
- **リーンキャンバス or 事業性メモ**: 競合・収益モデル・リスク。1枚で十分。

### 2.2 要件定義
- **要件定義書**: 機能要件はユーザーストーリー形式+**受け入れ条件(Given/When/Then)**。受け入れ条件はAIがそのままテストコードに変換できる。
- **ペルソナ(利用者像)**: 役割・主な業務・ITリテラシー・利用頻度・利用環境。画面設計と操作マニュアルの難易度を決める入力。
- **非機能要件一覧**: 性能、セキュリティ、可用性など。指定しないとAIは非機能を考慮しない。
- **用語集(ユビキタス言語)**: AIが生成するコードの命名の一貫性に直結する最重要資料。
- **業務フロー図・権限マトリクス**(業務システムの場合)

### 2.3 モックアップ作成
- **画面一覧・画面遷移図**(Mermaid推奨)
- **デザインの意思決定メモ**: Figmaリンクだけでは AI に渡せないため、要点をテキスト化する。
- **デザイントークン / design.md**: 色・タイポグラフィ・コンポーネント規約。

### 2.4 基本設計
- **アーキテクチャ図**(C4モデルのコンテキスト図+コンテナ図程度、Mermaidで)
- **ADR**: 技術選定の記録。1判断=1ファイル。**AI駆動開発で最も費用対効果が高い資料**。
- **ER図 / データモデル定義**
- **API仕様(OpenAPI / スキーマ定義)**: 機械可読な定義を正とする。
- **外部連携一覧・インフラ構成図・認証認可方式**

### 2.5 詳細設計
クラス図レベルの詳細設計書は**ほぼ不要**(AIが基本設計+規約から生成でき、すぐ陳腐化する)。代わりに:
- **CLAUDE.md / AGENTS.md(プロジェクト規約)**: ディレクトリ構成、命名規則、エラーハンドリング方針、テスト方針。AIへの常設指示書であり、実質これが詳細設計の代替。
- **複雑なビジネスロジックの仕様書のみ個別に**: 料金計算、状態遷移、締め処理など「コードを読んでも業務的正しさを判断できない」箇所に限定。

### 2.6 開発・テスト
- **テスト観点表・テスト計画**: 何を自動テストし、何を手動確認するか。
- **テストコード自体**: 受け入れ条件→テストコードが実質の仕様書。
- **PR説明文**: 変更の意図を書く。git履歴がAIの参照可能な文脈になる。
- **CI設定と品質ゲートの定義**(lint・型チェック・カバレッジ基準)

### 2.7 PoC
- **検証計画書**: 仮説、成功/失敗の判定基準、期限。
- **検証結果レポート**: 採用したもの・**捨てたものとその理由**。
- **「このコードは使い捨て」の明示**(README)

### 2.8 リリース
- **リリース手順書 / ロールバック手順**(Runbook形式。コマンドで書けばAIに実行させられる)
- **CHANGELOG**
- **環境変数一覧**(値ではなくキーと用途)、**監視・アラート設定一覧**

### 2.9 保守
- **運用Runbook**: 定期作業、問い合わせ対応、障害時の初動手順。
- **ポストモーテム(障害報告書)**: 原因・影響・再発防止。AIの障害調査の参照資料になる。
- **ADRの継続**: 保守フェーズの技術判断も同じ場所に追記。
- **依存関係更新ポリシー**

### 2.10 リポジトリ構成の例

```
docs/
├── standards/             # 標準・規程(案件をまたいで使う)
│   ├── ai-driven-dev-lifecycle.md  # 開発標準(正本)
│   ├── governance.md               # 規程(文書管理・記録・AI利用・フォーマット)
│   └── agents-roster.html          # チーム名鑑(編集しない一枚物)
└── project/               # 案件の書類(雛形を埋めて使う)
    ├── 01_prd.md          # 企画
    ├── 02_requirements.md # 要件定義(ストーリー+受け入れ条件)
    ├── 03_glossary.md     # 用語集
    ├── 04_architecture.md # 基本設計(構成図・ER図はMermaid)
    ├── 05_screens.md      # 画面一覧・遷移図
    ├── 06_test-plan.md    # テスト計画
    ├── 07_manual.md       # 操作マニュアル・リリース案内
    ├── adr/               # 技術判断の記録(1判断1ファイル)
    ├── poc/               # 検証計画と結果
    └── runbook/           # リリース・運用手順
CLAUDE.md                  # AI向け規約(実質の詳細設計)
```

> **要点**: 「AIに毎回口頭で説明していることをファイルにする」のがAI駆動開発のドキュメント戦略。企画書・要件・ADR・規約(CLAUDE.md)の4つが背骨で、それ以外は薄く保つ。

## 3. ISO規格への対応

ISO側の要求は「**何を記録し、どう管理するか**」であり、文書の形式(Word か Markdown か)は規定していない。Markdown + Git 構成はそのままマッピングできる。

### 3.1 資料とISO規格の対応表

| 資料 | 対応する主な規格 |
|---|---|
| ライフサイクル全体の枠組み | ISO/IEC/IEEE 12207(ソフトウェアライフサイクルプロセス) |
| 各文書の記載内容の定義 | ISO/IEC/IEEE 15289(ライフサイクル情報項目) |
| 要件定義書・受け入れ条件 | ISO/IEC/IEEE 29148(要求工学) |
| 非機能要件一覧 | ISO/IEC 25010(品質特性モデル) |
| アーキテクチャ図・ADR | ISO/IEC/IEEE 42010(アーキテクチャ記述) |
| テスト計画・観点表・結果 | ISO/IEC/IEEE 29119(ソフトウェアテスト文書) |
| Runbook・障害記録・アクセス管理 | ISO/IEC 27001(ISMS) |
| 文書管理・記録全般 | ISO 9001(QMS) |
| AI利用の統制 | ISO/IEC 42001(AIマネジメントシステム)、ISO/IEC 5338(AIシステムのライフサイクル) |

### 3.2 監査に耐えるための追加4点

1. **文書管理規程を1枚作る**
   「main ブランチのファイルが承認済み版」「PR承認が承認記録」「git tag が版数」「改訂履歴は git log」と明文化。PR承認+ブランチ保護が承認プロセスの証跡になる。
2. **トレーサビリティを付ける**
   要件に一意ID(`REQ-001` など)を振り、設計・テスト・PR がIDを参照する。トレーサビリティマトリクスはAIにリポジトリを走査させて自動生成できる。
3. **記録(records)の保管を決める**
   テスト結果(CIログ)、リリース承認、障害対応記録は保管期間を決めて残す。
4. **AI生成コードの統制を明文化する(ISO/IEC 42001)**
   「AI生成コードは必ず人間がレビューして承認」「レビュー証跡はPR承認」「AIに渡してよい情報の範囲」をポリシー化。

### 3.3 注意点

- ISO 9001 / 27001 が目的なら上記で実務に耐える。医療機器(IEC 62304)や自動車(ISO 26262)などの規制産業は様式・承認プロセスまで要求されるため、軽量アプローチのままでは不足。
- 12207 はプロセス規格であり「この文書を作れ」という規格ではないため、反復的な進め方と矛盾しない(反復開発への適用は ISO/IEC/IEEE 26515)。

## 4. TDD × AI駆動開発の流れ

核になる考え方: **テストが人間からAIへの発注書になる**。品質担保の重心が「実装コード」から「テストコード」に移る。

### 4.1 二重ループ構造

```
要件定義
  └─ ユーザーストーリー + 受け入れ条件(Given/When/Then)
        │
        ▼
【外側ループ】ストーリー単位(数時間〜数日)
  1. 受け入れ条件を「失敗する受け入れテスト」に変換
        │
        ▼
【内側ループ】関数・クラス単位(数分〜数十分)
  2. Red    : 失敗するユニットテストを1つ書く
  3. Green  : テストを通す最小限の実装を書く
  4. Refactor: テストを保ったまま整理する
  5. → 2に戻り、受け入れテストが通るまで繰り返す
        │
        ▼
  6. 受け入れテストが通ったらストーリー完了 → 次のストーリーへ
```

### 4.2 役割分担とガードレール

| ステップ | 主担当 | ポイント |
|---|---|---|
| テスト作成 | 人間(AIが補助) | テストは仕様そのもの。人間が精読して承認する |
| Red の確認 | 必須工程 | テストが実際に失敗することを確認してから実装へ。CLAUDE.md に明記 |
| Green(実装) | AI | **テストの修正による帳尻合わせを禁止**。「テスト変更が必要なら理由を報告して停止」と規約に明記 |
| Refactor | AI(人間がレビュー) | テストが安全網なので大胆に任せられる |

### 4.3 ハマりやすい点

- **AIのテスト・ハッキング**: テスト書き換えや期待値ハードコード。受け入れテストは別セッション・別レビューで検証する。
- **モック過多**: 「外部I/O以外はモックしない」を規約に書く。
- **テストの粒度が細かすぎる**: 「振る舞い(公開インターフェース)をテストし、実装をテストしない」が原則。
- **過剰適用**: モックアップ検証やPoCにTDDは向かない。要件が受け入れ条件まで落ちた開発フェーズに適用する。

## 5. テストレベル(単体・結合・総合)とライフサイクル

### 5.1 V字モデル: 設計とテストの対応

```
企画・要件定義 ──────────────────── 総合テスト(＋受入テスト)
     \                                    /
      基本設計 ──────────────── 結合テスト
           \                        /
            詳細設計 ──── 単体テスト
                 \          /
                   実装
```

- **要件定義 ↔ 総合テスト**: 要件どおりに動くかをE2Eで検証
- **基本設計 ↔ 結合テスト**: API・DB・外部連携のつなぎを検証
- **詳細設計 ↔ 単体テスト**: 個々の関数・クラスを検証

この対応関係はTDDでもAI駆動でも変わらない。変わるのは**テストをいつ書くか**。

### 5.2 シフトレフト(W字化): テスト設計を左側に移動

```
要件定義 ──── 受け入れ条件を書く ──→ ここで総合テストシナリオが決まる
基本設計 ──── OpenAPI/ER図を書く ──→ ここで結合テストの契約が決まる
実装     ──── Red-Green-Refactor ──→ 単体テストは実装と同時(内側ループ)
```

- 外側ループ = 総合テストレベル(Given/When/Then が E2E シナリオになる)
- 中間 = 結合テストレベル(OpenAPI・スキーマ定義が契約=期待値になる)
- 内側ループ = 単体テストレベル(Red-Green-Refactor)

### 5.3 時系列でのライフサイクル全体

```
1. 企画
2. 要件定義      … 受け入れ条件 → 総合テストシナリオ(この時点では全部Red)
3. モックアップ
4. 基本設計      … API仕様・ER図 → 結合テスト観点を確定
5. 開発イテレーション(ストーリー単位で繰り返し)
   ├─ a. ストーリーの受け入れテスト(E2E)を書く → Red
   ├─ b. 必要なAPIの結合テストを書く → Red
   ├─ c. 単体TDDで実装(Red→Green→Refactor を高速に回す)
   ├─ d. 結合テストが Green になる
   └─ e. 受け入れテストが Green になる → ストーリー完了
6. リリース前     … 総合テスト一式+非機能テスト(性能・セキュリティ)
7. リリース       … 受入テスト(UAT)
8. 保守          … 全テストが回帰テスト資産になる
```

**総合テストは「最後の工程」ではなく「最初に書かれて最後に全部通る」**。進捗が「総合テスト100本中70本Green」のように定量で見える。

### 5.4 テストピラミッド(量・タイミング・分担)

| レベル | 量の目安 | 実行タイミング | AIと人間の分担 |
|---|---|---|---|
| 単体 | 多(7割) | 毎コミット(秒単位) | AI主導で生成、人間はサンプリングレビュー |
| 結合 | 中(2割) | 毎PR(CI、分単位) | OpenAPI等からAI生成、人間は契約(仕様側)を承認 |
| 総合 | 少(1割) | ストーリー完了時+リリース前 | 受け入れ条件は人間が主導、E2Eコード化はAI |

総合テストを増やしすぎると遅くて不安定になり、AIの高速な開発ループを殺す。

## 6. テスト用DBの方針

原則: **テストレベルによって変える。「本番と違うDB製品で代用」は結合以上では避ける。**

| レベル | DB | データ |
|---|---|---|
| 単体 | 使わない(フェイク) | テスト内に直書き |
| 結合 | 本番と同一製品・同一版をコンテナで使い捨て | テストごとに投入・分離 |
| 総合 | 本番と同一構成のステージング | マスキング済みの本番相当データ |

### 6.1 単体テスト: DBを使わない

リポジトリ層をインターフェースで切り、インメモリのフェイクで代替する。SQLの正しさを検証したいテストは結合テストに分類して切り出す。

### 6.2 結合テスト: 本番と同じDB製品・同じメジャーバージョン

「本番はPostgreSQL、テストはSQLite」の代用はアンチパターン:

- SQL方言の差(UPSERT、JSON関数、ウィンドウ関数、照合順序)で「テストは通るのに本番で壊れる」
- 型の扱い・制約・トランザクション分離レベルの挙動差
- マイグレーションファイルが方言差で共用できない

Testcontainers(またはCIのDockerサービス)で本番同等のDBを数秒で起動できるため、代用する理由自体がほぼ消えた。スキーマは本番と同じマイグレーションファイルで作る。

### 6.3 総合テスト: 本番と同一構成のステージング

DB製品・バージョンだけでなく構成(接続プーリング、レプリカ、拡張機能)まで揃える。データ量による性能問題はここでしか見つからない。

### 6.4 データの分離(AI駆動では特に重要)

- 結合テスト: テストごとにトランザクションをロールバック、またはスイートごとにコンテナ/スキーマを使い捨てる。テスト間のデータ共有は順序依存の不安定テストを生む。
- **並列で動くAIエージェントにはインスタンス分離が必須**。ブランチ/ワークツリーごとに独立したコンテナを立てる(NeonのDBブランチングなども同目的)。共有開発DBは互いのテストを壊し合う。
- 総合テストで本番データを使う場合は**必ずマスキング**する(ISO/IEC 27001・プライバシー法制の典型的指摘事項)。
