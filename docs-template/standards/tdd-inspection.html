TDD FOR BEGINNERS 5/5 ・ STD-AIDEV-007

はじめてのTDD ⑤ 検収編

AIに書かせたテストの抜けを、人間はどう見つけるか。答えは「数を数える」です。

5-1

人間の仕事は「数の検収」

AIが書いた大量のテストコードを読んで網羅を確かめるのは無理です。そこで検収を「中身を読む」から「数を数える」に変えます。テストには必ず出どころの文書(正本)があり、正本からあるべき本数(母数)が決まります。母数と実際の本数が合っているかだけを見るのが人間の仕事です。

そのためにAIへの依頼を二段階に分けます。いきなりコードを書かせないのがコツです。

  1. 一覧表を出させる。「正本からテストケースの一覧表(ID・シナリオ・観点・レベル)を出して」AI
  2. 数と抜けだけ検収する。「US-03のシナリオが要件定義に4本あるのに表に3本しかない」を見つける。中身のレビューはまだしない人間
  3. 承認した表をコード化させ、全件Redを確認する。ここから先は②改修編の凍結ルールどおりAI
5-2

母数はどの文書から来るか

レベルごとに正本と母数が決まっています。総合と結合は一括生成+数の検収、単体は事前に一覧を作りません(①基本編の「1本ずつ」のリズムをAIにも守らせます)。

5-3

番号を名前に埋めると、検収が機械になる

受け入れ条件のシナリオに US-01-S1 形式の番号を振り、テスト名に必ず含めます。

NAMING RULEtest_US01_S2_在庫0の備品は借りられない
ストーリー番号+シナリオ番号+仕様文。この形にすると「S番号の欠番 = 未実装」が機械的に分かり、突き合わせをAIに任せられます: 「02の全シナリオと tests/ を突き合わせ、テストのないシナリオを列挙して」。実施はストーリー完了時とリリース前の2回です。

番号台帳を別に作るわけではない点が大事です。名前に埋めるだけなので、統制規程の「重いトレーサビリティは平時に維持しない」という方針と両立します。監査で対応表を求められたら、この名前からAIに生成させれば済みます。

5-4

外側と内側で、検収のやり方が違う

ループAIへの出力のさせ方人間の検収
外側(総合・結合)一括: 正本 → 一覧表 → 承認 → コード化数の突き合わせ(表 vs 母数)
内側(単体)逐次: 3拍子の中で1本ずつ毎回のRed確認(①基本編のリズム)

単体テストの一覧を事前に出させたくなったら、①基本編のつまずきポイント3を思い出してください。まとめて書かせると、どの失敗がどの原因か分からなくなります。AIに聞いてよいのは「次の1本」の材料だけです: 「このストーリーでまだテストしていない境界値は?」

5-5

卒業チェック: 3問

Q1. AIに総合テストを書かせるとき、最初に出させるものは?

テストケースの一覧表です。コードではありません。人間は表の数を母数(シナリオ数)と突き合わせ、承認してからコード化させます。

Q2. 結合テストの母数はどの文書から数える?

04 基本設計書です。エンドポイント数×代表ケース+スキーマの制約数。02の権限マトリクスと同時利用者数が補助入力になります。

Q3. 単体テストも一覧表で検収する?

しません。単体は内側ループで1本ずつ生まれ、検収は毎回のRed確認です。事前に一覧を作らせるのは「一度にたくさん書く」をAIにやらせることになります。
