本起案は最小限の統制を定める。既存のGit運用(mainブランチ保護・PRレビュー)を統制の基盤に流用し、新規の仕組みは増やさない。要件IDトレーサビリティ等の重い管理は導入しない。監査で要求された時点で個別に検討する。

① 文書管理ルール

目的: 設計書・手順書・規程類の版管理を、既存のGit運用だけで完結させる。

ルール

  • mainブランチ上のファイルを承認済み版とする。
  • 変更はPRを通す。PRのApprove(承認者1名以上)を承認記録とする。
  • 改訂履歴はgit logで管理する。コミットメッセージに変更理由を書く。
  • mainブランチにブランチ保護を設定し、レビュー承認者を指定ロールに限定する。

証跡の残し方: PRのApprove履歴とgit logを証跡とする。台帳や記録シートは別途作らない。

例外時の扱い: 緊急修正でレビューを待てない場合、mainへの直接pushを一時許可する。事後24時間以内にPRを起票し、承認を得る。承認の有無はPRコメントに残す。

② 記録の保管

目的: CIのテスト結果・リリース承認・障害記録を、監査時に参照できる状態で残す。

ルール

  • CIのテスト結果はCI基盤(GitHub Actions等)のログをそのまま保管する。
  • リリース承認はリリースPRのApprove履歴を承認記録とする。
  • 障害記録はIssueまたは障害報告ドキュメントに、発生日時・影響範囲・対応内容・再発防止策を記載する。
  • 保管期間: 要確認。

証跡の残し方: CIログ・PR履歴・Issueへのリンクを、リリースノートまたは月次報告にまとめる。

例外時の扱い: 保管期間が確定するまで、既存ログを削除しない。CI基盤のログ自動削除設定を確認し、削除がある場合は無効化するか外部保存を検討する。

③ AI利用ルール

目的: AI生成コードの品質と、AIへの情報提供範囲を統制する。

ルール

  • AI生成コードは必ず人間がレビューする。レビュー未了のコードはmainにマージしない。
  • レビュー実施の証跡はPR承認とする。追加の承認フローは設けない。
  • 社外秘情報・個人情報をAIツールに入力してよい範囲: 要確認。
  • 範囲確定までは、顧客の個人情報・未公開の契約情報・認証情報(APIキー・パスワード)を入力対象から除外する。

証跡の残し方: PRのApprove履歴を証跡とする。

例外時の扱い: 入力の可否に迷う情報は入力せず、上長に個別確認する。

④ フォーマット規則

目的: 正本を1つに保ち、二重管理とAIが編集できない形式の混入を防ぐ。

ルール

  • 正本はテキスト形式に限る。文書=Markdown、図=Mermaid(Markdown内に埋め込み)、API仕様=OpenAPI(YAML)、DBスキーマ=マイグレーションファイル。
  • HTML・PDFは正本から作る生成物とする。生成物を直接編集しない。
  • 育て続ける文書はMarkdownを正本とし、配布形を生成する。以後編集しない一枚物はHTMLのみでよい。
  • ExcelやWordで受け取った資料は入力として扱い、内容をMarkdownに起こして正本化する。受領ファイル自体は正本にしない。
  • 置き場所: 標準・規程は docs/standards/、案件の書類は docs/project/ に置く。

証跡の残し方: 正本の改訂はPRで行い、git logを改訂履歴とする(①文書管理ルールに従う)。

例外時の扱い: 社外から様式指定(Excel提出等)がある場合のみ、生成物としてその様式に書き出す。編集は正本側で行う。
