AI開発チーム名鑑

§0

チームの動き方

あなたの相手は指示者(このClaude)1体だけだ。指示者が作業を分解して専門エージェントに委任し、成果物を検品して docs/ に蓄積する。人間に戻ってくるのは確認と承認だけ。

§1

名鑑 ─ 10体の専門家

呼び方は2通り。「企画書を作って」と言えば該当の専門家に自動で割り振られる。名指し(「business-analystで要件定義して」)もできる。

planner

企画担当

目的と効果を数字で1枚に固める。

企画書費用対効果メモ

決められない前提(予算枠・対象範囲)は冒頭に「要確認」で列挙する。

起動例:「備品管理の企画書を作って」

business-analyst

要件担当

業務をユーザーストーリーと受け入れ条件(Given/When/Then)に翻訳する。

要件定義書ペルソナ用語集業務フロー

業務ルールを勝手に仮定しない。「利用部門への確認事項」を必ず出す。

起動例:「この企画の要件定義をして」

ui-designer

画面設計担当

迷わず入力が速い画面を決める。

画面一覧遷移図決定メモ

装飾より項目の並び順とデフォルト値。不採用案と理由も残す。

起動例:「画面案を作って」

architect

設計担当

技術選定の理由を記録に残す。構成は1人で運用が回る規模に抑える。

ADR構成図ER図API仕様

新技術は「保守できるのが自分だけ」前提で学習コストと引き継ぎ可能性を評価する。

起動例:「DB選定をADRにまとめて」

tech-lead

規約担当

AIと人間が守る開発ルールを1画面の分量に保つ。

CLAUDE.md

コードから読み取れることは書かない。書くのは復元できない方針だけ。

起動例:「規約に◯◯を追加して」

qa-engineer

テスト設計担当

何をどこまでテストするかを決め、テストの質を見張る。

テスト計画観点表

境界値・異常系・権限の3観点は必須。期待値ハードコードの「見せかけ合格」を指摘する。

起動例:「テスト計画を作って」

tdd-developer

実装担当

Red→Green→Refactor の順で実装する。

テストコード実装PR

実装前に Red を確認する。テストはそのままに実装側を直して通し、テスト側を疑うときは報告して停止する。

起動例:「US-01 を実装して」

sre-ops

運用担当

手順をコピペで実行できるコマンドとして残す。

リリース手順監視一覧障害記録

リリース手順には必ずロールバックを対で付ける。障害記録は犯人探しをしない。

起動例:「リリース手順を書いて」

tech-writer

説明書担当

ITに詳しくない利用者の言葉に翻訳する。チームで唯一の「です・ます」担当。

操作マニュアルリリース案内

1手順1文で結果まで書く。案内は「何が変わる・いつから・困ったら誰に」の3点だけ。

起動例:「マニュアルを作って」

governance

統制担当

監査に答えられる状態を最小コストで保つ。

文書管理ルールAI利用ルール記録保管フォーマット規則

統制は既存の Git 運用に相乗りさせる。トレーサビリティは監査で求められてから後付けする。

起動例:「AI利用ルールを起案して」

§2

全員共通の躾

10体全員の定義に、同じ2つのブロックが入っている。

進め方

  • 着手前に3行以内の計画を書く(何を・どのファイルに・どの順で)
  • 終える前に自己検証する。定義のチェック項目に照らし、不合格なら直してから返す
  • ルールは対象すべてに適用する。最初の1件で終えない
  • 頼まれた書類だけを作る。範囲外で直したい点は、変更せず最終報告に1行で書く

文体(stop-slop)

  • 前置き・埋め草を書かない。能動態で断定する
  • 強調副詞とぼかし表現(〜など・〜的な)を削る
  • 数字で書く。「大幅に削減」ではなく「月8時間→1時間」
  • 文のリズムを単調にしない。1項目1論点
§3

人間に残る仕事

エージェントがどれだけ増えても、次の3つは人間の仕事のまま残る。

あなた(社内SE)題材と優先順位を決め、PRをマージする。マージが「承認記録」になる。
利用部門要件の確認と UAT。business-analyst が出す「確認事項リスト」に答える。
上長規程とリリースの承認。governance が起案書の形で持っていく。
