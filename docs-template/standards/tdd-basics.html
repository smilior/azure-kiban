TDD FOR BEGINNERS 1/5 ・ STD-AIDEV-007

はじめてのTDD ① 基本編

テストは「答え合わせ」ではなく「発注書」です。先に書く理由から始めます。

1-1

なぜテストを先に書くのか

TDD(テスト駆動開発)は、「こう動いたら合格」というテストを先に書き、それを通すために実装する進め方です。テストは完成後の答え合わせではなく、作る前に渡す発注書にあたります。

発注書が先にあると、2つのことが起きます。第一に、完成の定義がぶれません。実装者(私たちの場合はAI)は「テストが通ったら完了」という明確なゴールに向かって進みます。第二に、後から書くテストにありがちな「今の実装をなぞっただけの追認テスト」ができません。仕様から書いたテストだけが、実装の間違いを見つけられます。

1-2

3拍子: Red → Green → Refactor

TDDの1サイクルは3拍子です。数分単位でこれを繰り返します。

RED

まず失敗させる

失敗するテストを1本だけ書き、実行して失敗を見届けます。問題用紙を先に作るイメージです。最初から通るテストは、何も検証していません(問題を出す前に丸を付けるのと同じです)。

GREEN

最小の実装で通す

そのテストを通すいちばん小さい実装を書きます。きれいさは後回しで構いません。ここで触っていいのは実装だけで、テストには触りません。

REFACTOR

きれいにする

テストが通ったまま、重複を消し名前を整えます。テストが安全網になっているので、思い切って整理できます。ここでもテストは触りません。

1-3

付箋とチェック: ストーリーとシナリオ

旗の話に入る前に、言葉を2つだけ覚えます。システムで「できること」を1つずつ付箋に書き出したもの(備品を借りられる・返却できる…)がストーリーで、US-01、US-02と番号を振ります。付箋1枚が開発の単位で、2〜3日で作れる大きさに切ります。

次に付箋を1枚取り、「どうなったら合格?」を書き出します。ふつうに返したら在庫に戻る、遅れて返したら延滞記録が残る…。この合格チェック1個1個がシナリオ(US-03-S1、S2…)で、チェック1個 = E2Eテスト1本になります。番号は住所と同じです: US-03-S2 は「3丁目の2番地」。

確認クイズ「備品を予約できる」と「予約が重複したらエラーになる」、ストーリーはどちら? — 前者です。後者はそのストーリーの合格チェック(シナリオ)になります。
1-4

ゴールの旗: E2Eテストを最初に立てる

機能(ストーリー)に着手したら、3拍子を回し始める前にE2Eテスト(画面から操作する総合テスト)を1本書いて「旗」を立てます。この旗は最初は当然Redで、すぐ通そうとはしません。単体テストの3拍子を何周も回し、実装が積み上がると、最後に旗がひとりでにGreenになります。それが「この機能は完成した」の合図です。

用語 ─ GIVEN / WHEN / THEN旗の中身は、要件定義で決めた「受け入れ条件」です。受け入れ条件は3行で書きます: Given = 前提(〜の状態で) / When = 操作(〜したら) / Then = 結果(〜になるはず)。例:「Given 在庫0の備品がある / When 借りるボタンを押す / Then エラーが表示され、貸出は登録されない」。この3行がそのままE2Eテストの設計図になります。
1-5

つまずきポイント3つ

PITFALL 1テストを後から書く。それはTDDではなく「追認」です。実装の癖ごと正解扱いしてしまうため、間違いを見つけられません。
PITFALL 2Redの確認を飛ばす。書いたテストがいきなり通ったら喜ぶところではなく、疑うところです。テスト自体が壊れているか、何も検証していない可能性があります。
PITFALL 3テストを一度にたくさん書く。1本書いて1本通す、が基本のリズムです。まとめて書くと、どの失敗がどの原因か分からなくなります。例外はゴールの旗(E2E)の1本だけで、あれは意図して赤のまま置きます。赤を待たせてよい本数はレベルで違います(③レベル編で説明します)。
