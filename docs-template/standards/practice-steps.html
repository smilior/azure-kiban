実践ガイド ② 手順編
─ コピペ運転マニュアル

1ストーリーで打つプロンプトは4つです。そのままコピペで使えます。

2-1

プロンプト4つ(US-03 備品を返却する の例)

PROMPT 1

着手して、表を出させる

駅1→2
US-03(備品を返却する)に着手して。
02_requirements.md の受け入れ条件から、テストケース一覧表
(ID・シナリオ・観点・レベル)を先に出して。コードはまだ書かないで。

打つと:ストーリーIssue(雛形: .github/ISSUE_TEMPLATE/story.md)とブランチが作られ、一覧表がIssueのコメントに返ってきます。「コードはまだ書かないで」の一文が二段階出力の鍵です。

PROMPT 2

数を検収して返事する

駅3(あなた)
US-03-S3(延滞中の備品を返す)が表に抜けてる。追加して。

見るのは数だけ:02_requirements.md のシナリオ数と表の行数を突き合わせます。合っていれば次の一言で発車します。

OKの返事: OK、この表で確定。進めて。/ 観点の追加: 返却の二重送信のケースも足して。

PROMPT 3

旗立てから実装まで一括で任せる

駅4→6
確定した表のとおりE2Eと結合テストをコード化し、全件Redを確認して。
そこからテストは凍結し、内側ループ(単体TDD)で実装を進めて。
旗が全部Greenになったら、突き合わせ結果(US-03の欠番なし)と一緒に報告して。

打つと:ここが最長区間ですが放置で大丈夫です。Red確認の徹底・テスト書き換えの禁止・1本ずつのリズムは CLAUDE.md に入っているので、毎回書く必要はありません。

PROMPT 4

PRにさせる

駅6→7
PRを作って。説明には変更の意図と、テストの実行結果を貼って。

打つと:PRが届きます。あとは下のチェックリストで見てマージすれば、このストーリーは完了です。

2-2

PRレビューの急所3点

全行を読む必要はありません。この3点だけ確認できればマージできます。

  • テストの差分が、検収した表のとおりか。表にないテストが増えていたり、あるはずのS番号が欠けていたら差し戻し
  • 握りつぶしがないか。空のエラー処理・TODOコメント・スキップされたテスト(skip/only)は「動いているふり」の温床
  • 頼んでいない変更が混ざっていないか。US-03のPRに無関係なファイルの変更があったら戻させる
2-3

詰まったときの対処

実装区間(駅5)で起きがちな3つの症状と、返す一言です。

2-4

運転のコツ3つ

TIP 11ストーリー = 1ブランチ = 1セッション。セッションを長く引きずると文脈が濁ります。2〜3日で旗がGreenにならないサイズなら、ストーリーの切り方(要件定義)に戻って分割します。
TIP 2迷ったら止めるのはあなたの権利。「一旦止めて。現状を要約して」はいつ打っても安全です。テストが凍結されている限り、途中で止めても壊れるものはありません。
TIP 3同じ指摘を2回したら、規約に1行足す。「またTODOが残ってる」が2回続いたら、CLAUDE.md に「TODOを残さない。未対応は報告に書く」を追記します。指摘は消耗品、規約は資産です。
