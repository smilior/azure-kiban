TDD FOR BEGINNERS 3/5 ・ STD-AIDEV-007

はじめてのTDD ③ レベル編

単体・結合・総合。3つのテストで原則は1つ、違うのはリズムだけです。

3-1

3レベルとも原則は同じ

改修編の鉄則「テストを先に動かして Red → 凍結 → 実装で Green」は、どのレベルでも同じです。違うのはテストを動かす「きっかけ」と、1周のリズムだけです。

レベルテストを動かすきっかけ1周のリズム
総合(E2E)受け入れ条件の変更。業務の正解が変わったときストーリー単位(数時間〜数日)
結合契約の変更。API仕様・DBスキーマが変わったとき(部署カラム追加なら、マイグレーションと結合テストの期待値が先)ストーリーの中(数十分〜)
単体次に作る1つの振る舞い。テスト1本→実装→また1本と交互に数分

単体は交互に書くので「同時にいじっている」ように見えますが、1サイクルの中では同じ規律です。テストを1本書いたら凍結し、実装だけでGreenにする。Refactor中はテストに触りません。

3-2

Redのまま置いていいのは何本?

基本編の「1本書いて1本通す」の正確な意味は、「いま通そうとしているテストは常に1本」です。存在していい赤が1本、ではありません。赤のまま待たせてよい本数と時間は、レベルごとに違います。

レベルRedのまま置いていい本数Greenまでの猶予
総合(E2E)1本だけ(着手中ストーリーの旗)数日。赤のまま置くのが役目
結合着手中の契約の分だけ(少数)数十分〜数時間
単体常に1本数分。ここが「1本書いて1本通す」の主戦場

理由は、範囲が広いテストほど、失敗の原因を教えてくれないからです。赤いE2Eから分かるのは「どこかが未完成」だけで、どの関数が悪いかは分かりません。だから原因を特定できる単体でリズムを刻み、E2Eは1本だけ立てて進捗の物差しに使います。テストピラミッドの本数比(単体7:結合2:総合1)は、この理屈の帰結です。

例えるなら、E2Eは月間目標(1つだけ壁に貼り、達成まで剥がさない)、結合は今週の目標(数個)、単体は今日のToDo(1個ずつ消す)。目標を10個貼る人が迷子になるのと同じで、赤いE2Eを並べるとノイズになります。

3-3

唯一の例外: リファクタで単体テストが壊れたとき

振る舞いを何も変えていないのに、コード整理で単体テストが落ちることがあります。これは仕様変更ではなく、「テストが実装の中身に依存しすぎている」というテスト側の欠陥のサインです。このときだけ、テストを振る舞いベース(公開インターフェースだけを見る形)に書き直すのが正当です。

E2Eと結合は画面・APIという外側しか見ないため、この問題はほぼ起きません。規約の「振る舞いをテストし、実装をテストしない」は、この事故を予防するためにあります。

3-5

クリックすると答えが開きます。3問とも即答できたら、このシリーズは卒業です。

Q1. Green にするために触っていいのは、テスト? 実装?

実装だけです。Redを確認した瞬間からテストは凍結され、「答え合わせの基準」に変わります。テストを触りたくなったら図1のフローへ。

Q2. 改修(仕様変更)が来たとき、最初に動かすのはどっち?

テスト側です。受け入れ条件を更新→テストを新仕様に書き換え→Redを確認してから、実装に手を付けます。実装から直し始めると、完了の基準がないまま作ることになります。

Q3. リファクタ中に単体テストが壊れました。どうする?

振る舞いを変えていないなら、テストが実装の中身に依存しているサインです。振る舞いベースに書き直します(これは唯一の例外)。振る舞いを変えていたなら手順違反なので、実装を戻してテストから先にやり直します。
