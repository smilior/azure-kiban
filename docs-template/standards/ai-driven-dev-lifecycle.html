§0

基本原則

AI がコードを書く。人間が残すのは意図と判断根拠だ。原則は3つ。

Markdown で Git 管理

docs/ をコードと同じリポジトリに置く。AI がそのまま読めるため、仕様書が毎回のプロンプトの代わりになる。Word と Excel はこの用途に使わない。

What より Why

コードの動作は AI が読み取れる。復元できないのは「なぜ選んだか」「なぜ捨てたか」。ADR(Architecture Decision Record)に 1判断 = 1ファイルで蓄積する。

生成できるものは生成

API 仕様は OpenAPI 定義を正とし、詳細設計はテストコードと型に語らせる。更新しないと決めた文書は消す。陳腐化した文書は AI のコンテキストを汚染する。

§1

フェーズ別の成果物

背骨は4つ。企画書・受け入れ条件付きの要件・ADR・CLAUDE.md。この4つを常に最新に保ち、残りは薄く保つ。

★ = 背骨の4文書

01企画 企画書費用対効果メモ 目的・対象業務・効果見込み(工数削減など)・スコープ外の明記。以降すべてのフェーズで AI に渡す前提資料になる。
02要件定義 要件定義書+受け入れ条件ペルソナ非機能要件一覧用語集業務フロー・権限 受け入れ条件は Given / When / Then で書く。AI がそのままテストコードに変換できる。ペルソナ(役割・ITリテラシー・利用頻度)が画面とマニュアルの難易度を決める。用語集は生成コードの命名の一貫性に直結する。
03モックアップ 画面一覧・遷移図意思決定メモ 利用部門と合意した画面案が証跡になる。決めた理由・不評だった案はテキストで残す。ツールは問わない。
04基本設計 ADR構成図ER図API仕様インフラ・認証 ADR は AI 駆動開発で最も費用対効果が高い資料。図は Mermaid、API がある場合は機械可読な定義(OpenAPI)を正とする。
05詳細設計 CLAUDE.md(規約)複雑ロジック仕様のみ クラス図レベルの設計書は書かない。規約とテストコードが代替する。個別仕様は料金計算・状態遷移など「コードを読んでも業務的正しさを判断できない」箇所に限る。
06開発・テスト テスト計画・観点表テストコード=仕様PR 説明文CI・品質ゲート PR に変更の意図を書く。git 履歴がそのまま AI の参照できる文脈になる。
07PoC・トライアル 検証計画結果レポート使い捨て宣言 新ツール・新技術の試行。捨てた案と捨てた理由を残す。残さないと半年後に同じ検証を繰り返す。試行コードは本番に持ち込まない。
08社内リリース Runbook・ロールバック操作マニュアル・利用者案内設定・監視一覧 手順をコマンドで書けば AI に実行させられる。利用部門への案内と教育までがリリース。マニュアルの下書きは AI に画面仕様から作らせる。
09保守 運用 Runbook障害記録引き継ぎ = docs/ 一式ADR 継続 社内SEの最大リスクは属人化。専用の引き継ぎ資料を別に作るのではなく、この docs/ 一式を読めば引き継げる状態を保つ。障害記録の蓄積は AI の障害調査の参照資料になる。

§2

監査・統制への対応

会社が ISO 9001 / 27001 の認証を持っていても、要求されるのは「何を記録し、どう管理するか」であって文書の形式ではない。Markdown + Git のままで足りる。守るのは次の4点だけ。

GOV-01

文書管理のルールを1枚作る

「main のファイルが承認済み版。PR 承認が承認記録。git log が改訂履歴。」と明文化する。ブランチ保護で承認者を限定すれば、PR がそのまま承認の証跡になる。監査で「どれが正の版か」に即答できる状態を保つ。

GOV-02

記録を残す

テスト結果(CI ログ)・リリース承認・障害対応記録は保管期間を決めて残す。変更管理の証跡は PR 履歴がそのまま使えるので、IT統制(内部統制)の変更管理への回答にもなる。

GOV-03

AI 利用のルールを決める

AI 生成コードは人間がレビューして承認する。証跡は PR 承認。社外秘・個人情報を AI に渡す範囲も決めておく。情シスは全社の AI 利用ルールを聞かれる立場でもある。まず自分の開発で運用実績を作っておくと答えやすい。

GOV-04

フォーマット規則を決める

正本はテキスト形式に限る: 文書=Markdown、図=Mermaid、API仕様=OpenAPI。HTML・PDF は正本から作る生成物として扱い、編集は正本側で行う。Excel・Word で受け取った資料は Markdown に起こして正本化する。置き場所は docs/standards/(標準・規程)と docs/project/(案件書類)。

やりすぎ注意:要件 ID(REQ-001)によるトレーサビリティマトリクスは、監査で実際に求められてからで足りる。AI にリポジトリを走査させれば後付けで生成できるので、平時から人手で維持しない。

§3

TDD の進め方

テストが人間から AI への発注書になる。品質担保の重心は実装コードからテストコードへ移る。ストーリー単位の外側ループと、関数単位の内側ループの二重構造で回す。

役割分担とガードレール

ステップ主担当ルール
テスト作成人間(AI が補助)テストは仕様そのもの。人間が精読して承認する。観点の網羅(境界値・異常系・権限)は AI にレビューさせる。
Red の確認必須工程テストが実際に失敗することを確認してから実装に進む。CLAUDE.md に明記する。最初から通るテストは何も検証していない。
Green(実装)AIテストはそのままに、実装側を直して通す。テスト側の誤りを疑うときは理由を報告して停止、と規約に明記する。
RefactorAI(人間がレビュー)テストが安全網になるため、重複排除や命名整理は大胆に任せられる。

ハマりやすい点

  • テスト・ハッキング。期待値のハードコードやテスト書き換えで「通す」。受け入れテストは別セッション・別レビューで検証する。
  • モック過多。モックがモックを呼ぶだけのテストができやすい。外部 I/O 以外はモックしない、と規約に書く。
  • 粒度が細かすぎる。プライベート関数まで縛るとリファクタのたびに壊れる。振る舞い(公開インターフェース)をテストし、実装をテストしない。
  • 過剰適用。モックアップ検証や PoC に TDD は向かない。要件が受け入れ条件まで落ちた開発フェーズにだけ適用する。
§4

テストレベルと V字モデル

単体・結合・総合の対応関係は V字モデルのまま。変わるのはテストを書く時期で、各レベルのテストを対応する設計工程まで前倒しする(シフトレフト)。

開発イテレーション(ストーリー単位)

  1. 主要業務フローの受け入れテスト(E2E)を書くRED
  2. 必要な API の結合テストを書くRED
  3. 単体 TDD で実装する(内側ループを高速に回す)
  4. 結合テストが通るGREEN
  5. 受け入れテストが通るGREEN = ストーリー完了。リリース前に総合一式と利用部門の受入確認(UAT)。
§5

テスト用 DB の方針

レベルによって変える。「本番と違う DB 製品で代用」は結合以上では禁止。

レベルDBデータ
単体使わない(インメモリのフェイク)テスト内に直書き
結合本番と同一製品・同一版をコンテナで使い捨てテストごとに投入・分離
総合本番と同一構成のステージングマスキング済みの本番相当データ

SQLite 代用がアンチパターンである理由

SQL 方言の差(UPSERT・JSON 関数・ウィンドウ関数・照合順序)、型の緩さ、トランザクション分離レベルの違いで、「テストは通るのに本番で壊れる」が起きる。結合テストで検証したいものが検証できない。Testcontainers か CI の Docker サービスで本番同等の DB を数秒で起動できる今、代用する理由は消えた。スキーマは本番と同じマイグレーションファイルで作る。

インスタンスの分離

  • テスト間でデータを共有しない。テストごとにロールバックするか、スイートごとにコンテナ / スキーマを使い捨てる。共有すると順序依存の不安定テストになる。
  • 並列 AI エージェントには分離が必須。共有の開発 DB は互いのテストを壊し合う。ブランチ / ワークツリーごとに独立したコンテナを立てる。
  • 本番データはマスキングしてから使う。社員・顧客の個人情報を含む本番データのテスト環境持ち込みは、ISMS 監査と個人情報保護の典型的な指摘事項。
