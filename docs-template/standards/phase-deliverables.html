§1

フェーズ別の成果物と完了条件

各書類の「詰める中身」は雛形の見出しと一致している。リンク先の雛形には記入ガイドと記入例が入っている。

01

企画

planner

作る価値があるかを、1枚で判断できる状態にする。

企画書01_prd.md
目的 ・ 対象業務と利用部門 ・ 現状の課題 ・ 効果見込み(作業時間×頻度×人数の数字) ・ スコープ外 ・ 概算費用 ・ 要確認事項
効果が数字で示され、あなた(と上長)が GO を出せる
02

要件定義

business-analyst

業務を、テストに変換できる条件に翻訳する。

要件定義書02_requirements.md
ペルソナ(役割・主な業務・ITリテラシー・利用頻度・利用環境) ・ ユーザーストーリー(「なぜ」付き) ・ 受け入れ条件(Given/When/Then、正常系+異常系) ・ 非機能3点(同時利用者数・バックアップ・権限マトリクス) ・ 業務フロー(Mermaid) ・ 利用部門への確認事項
用語集03_glossary.md
用語 ・ 定義 ・ コード上の名前(生成コードの命名を固定する)
利用部門が確認事項に回答済み。受け入れ条件がそのままテストになる粒度
03

モックアップ

ui-designer

画面の迷いを、実装前に潰す。

画面設計05_screens.md
画面一覧(ID・画面名・目的・主要項目・利用者) ・ 遷移図(Mermaid) ・ 決定メモ(不採用案とその理由) ・ 利用部門レビュー観点
利用部門が画面案に合意している
04

基本設計

architect

技術判断を、理由ごと記録に残す。

基本設計書04_architecture.md
構成図 ・ ER図 ・ 認証・権限方式 ・ インフラ構成 ・ 外部連携(図はすべて Mermaid)
ADRadr/NNNN-slug.md
背景 ・ 選択肢 ・ 決定 ・ 理由 ・ 捨てた案とその理由(1判断=1ファイル。API があれば OpenAPI 定義もここで正本化)
結合テストの契約(API仕様・スキーマ)が決まっている
05

詳細設計 = 規約

tech-lead

AIへの常設指示を1画面に保つ。クラス図は書かない。

開発規約CLAUDE.md
命名規則 ・ ディレクトリ構成 ・ エラーハンドリング方針 ・ テスト方針(7:2:1・DB方針) ・ TDDガードレール3点 ・ スコープ規律
複雑ロジック仕様必要時のみ
料金計算・状態遷移・締め処理など、コードを読んでも業務的な正しさを判断できない箇所だけ個別に書く
エージェントが迷わず実装に入れる(規約を読めば書き方が一意に決まる)
06

開発・テスト

qa-engineer / tdd-developer

Red→Green→Refactor でストーリーを1本ずつ積む。

テスト計画06_test-plan.md
レベル別配分(単体7割・結合2割・総合1割) ・ 観点表(境界値・異常系・権限) ・ テスト用DB方針 ・ UATの進め方
テストコード+実装+PRsrc/ ・ tests/
受け入れ条件から起こしたテストが実質の仕様書 ・ PR説明に変更の意図(なぜ) ・ CIのテスト結果が記録になる
ストーリーの受け入れテストが Green(テストの書き換えなしで)
07

PoC・トライアル(随時)

指示者

技術選定の迷いを、実測で潰す。順番を持たず必要なときに差し込む。

検証記録poc/NNNN-対象名.md
仮説 ・ 判定基準(数字で先に決める) ・ 期限 ・ 結果(実測値のみ) ・ 採否と理由(捨てた理由は将来への申し送り)
採否が数字で判定され、試行コードを本番に持ち込んでいない
08

社内リリース

sre-ops / tech-writer

利用部門が使える状態まで届ける。案内と教育までがリリース。

リリース手順runbook/release.md
前提条件 ・ 手順(コピペで実行できるコマンド+期待される出力) ・ 確認手順 ・ ロールバック手順(必ず対で)
マニュアル+案内07_manual.md
業務の流れ順の章立て ・ 1手順1文で結果まで ・ 案内は「何が変わる・いつから・困ったら誰に」の3点(唯一の「です・ます」文書)
UAT完了 ・ 上長承認 ・ 利用部門への案内済み
09

保守

sre-ops / architect

次の担当者(未来の自分)に、docs/ だけで引き継げる状態を保つ。

障害記録runbook/incident-*.md
事象 ・ 影響範囲 ・ 時系列 ・ 原因 ・ 再発防止(犯人探しではなく仕組みの改善)
ADRの継続+docs/の手入れadr/ ・ docs/ 全体
保守期の技術判断も同じ ADR に追記 ・ 改修は要件定義(02)に戻って回す ・ 引き継ぎ資料は作らず、docs/ 一式がそれを兼ねる
docs/ を読めば引き継げる状態が保たれている
